OpenAI LLM 경제성 쉬운 시나리오 리포트

0. 강의 목표

이 문서는 OpenAI 같은 LLM 서비스가 왜 단순한 소프트웨어가 아니라 GPU, HBM, 데이터센터, 구독 매출이 결합된 거대한 경제 시스템인지 설명하기 위한 강의 자료다.

수업이 끝나면 다음 네 가지를 이해하는 것이 목표다.

  1. LLM의 파라미터가 왜 GPU/HBM 비용으로 연결되는가
  2. 학습비보다 추론비가 왜 더 큰 반복 비용이 되는가
  3. 무료 사용자, 유료 구독, API 매출이 어떤 식으로 비용을 회수하는가
  4. 이 구조가 왜 DRAM/HBM 업황과 직접 연결되는가

1. 한 장으로 보는 OpenAI 경제 구조

OpenAI의 frontier 모델은 파라미터 수, 학습 데이터량, 학습 GPU 수, 실제 추론 원가를 공개하지 않는다. 따라서 아래 숫자는 OpenAI 내부 실측값이 아니라 공개 자료와 합리적 가정을 이용한 시나리오다.

핵심 구조는 간단하다.

쉽게 말하면, 사용자가 질문을 할 때마다 실제로 GPU와 HBM이 일을 한다. 그래서 OpenAI의 원가 구조는 일반 SaaS보다 반도체/클라우드 제조업에 가깝다.

2. 먼저 용어부터 정리

용어 쉬운 뜻 왜 중요한가
Parameter 모델 안에 저장된 숫자 weight 많을수록 HBM에 올릴 데이터가 늘어난다
Token 모델이 읽고 쓰는 글자 조각 요청이 많을수록 token/day가 늘고 추론비가 커진다
HBM GPU 옆에 붙는 초고속 메모리 모델 weight와 KV cache를 담고 계속 읽고 쓴다
KV cache 긴 대화 맥락을 유지하기 위한 임시 메모리 context가 길수록 HBM 사용량이 커진다
MoE Mixture of Experts, 일부 expert만 활성화하는 구조 계산량은 줄이지만 전체 weight는 저장해야 한다
Active params token 하나를 만들 때 실제 계산에 쓰는 파라미터 학습/추론 계산량을 추정할 때 쓴다
Total params 모델 전체에 존재하는 파라미터 HBM 용량을 추정할 때 중요하다
WAU Weekly active users, 주간 활성 사용자 서비스 규모를 나타낸다

핵심은 이것이다.

Total parameters → HBM 용량 부담
Active parameters × token 수 → 계산량 부담
사용자 수 × request 수 × token 수 → 매일 반복되는 추론비

3. 파라미터가 비용으로 바뀌는 과정

LLM의 파라미터는 모델 내부의 weight다. 쉽게 말하면 문장을 이해하고 다음 token을 예측하기 위해 학습된 숫자다.

파라미터 수가 크다
→ 저장해야 할 weight가 많다
→ HBM 메모리가 많이 필요하다
→ token 하나를 만들 때 읽어야 할 데이터가 많다
→ 추론 비용이 커진다
Precision Parameter당 용량 100B 모델 weight 의미
BF16/FP16 2 bytes 200GB 품질은 좋지만 메모리 부담이 큼
FP8 1 byte 100GB serving에서 많이 쓰는 절충
4-bit 0.5 byte 50GB 메모리는 줄지만 품질/운영 trade-off

100B 모델은 BF16으로는 200GB가 필요하다. H100 80GB 한 장에는 올라가지 않는다. 그래서 HBM 용량이 큰 GPU, 여러 GPU를 묶는 네트워크, 더 효율적인 quantization이 모두 중요해진다.

4. OpenAI에서 공개된 것과 비공개인 것

항목 공개 여부 현재 상태
GPT-5/GPT-5.x 파라미터 수 비공개 OpenAI가 total/active parameter를 공개하지 않음
GPT-5/GPT-5.x 학습 token 수 비공개 공개되지 않음
GPT-5/GPT-5.x 학습 GPU 수 비공개 공개되지 않음
GPT-5/GPT-5.x inference 원가 비공개 공개되지 않음
gpt-oss-120b 파라미터 공개 117B total, 약 5.1B active
gpt-oss-120b 메모리 공개 단일 80GB GPU에서 실행 가능하도록 설계
ChatGPT weekly active users 공개/보도 2026년 2월 기준 900M WAU 보도
ChatGPT paid subscribers 보도 2026년 초 50M paid subscribers 보도
OpenAI revenue run-rate 보도 2026년 초 월 $2B, 연환산 $24B 수준 보도

따라서 이 문서의 숫자는 정확한 내부값이 아니라 경제성 감각을 잡기 위한 시나리오다.

5. 모델 크기 시나리오와 HBM 요구량

OpenAI GPT-5 계열의 실제 크기는 비공개다. 그래서 네 가지 모델 크기 시나리오를 둔다.

시나리오 모델 구조 Total params Active params/token BF16 weight FP8 weight H100 80GB 최소 장수(BF16)
A. 공개 소형 기준 gpt-oss-120b급 MoE 117B 5.1B 234GB 117GB 3장
B. Frontier 효율형 MoE 500B 50B 1,000GB 500GB 13장
C. Frontier 대형 MoE 1.0T 100B 2,000GB 1,000GB 25장
D. Mega frontier MoE 2.0T 200B 4,000GB 2,000GB 50장

읽는 법은 단순하다.

  • HBM 용량은 total params 기준으로 본다.
  • 계산량은 active params 기준으로 본다.
  • MoE는 계산은 줄이지만, 전체 expert weight는 어딘가에 올라가 있어야 한다.

6. 학습비: 한 번 크게 쓰는 비용

LLM 학습 compute는 단순화하면 다음과 비례한다.

Training FLOPs ≈ 6 × active parameters × training tokens

가정:

  • H100 한 장의 실효 학습 성능: 400 TFLOP/s
  • H100 cloud-equivalent 비용: $3.5/GPU-hour
  • 실제 all-in 비용은 데이터 준비, 실험, failed run, RL/post-training, 인건비, 전력, 네트워크, 스토리지까지 포함
시나리오 Active params Token H100 GPU-days 50K H100 기준 기간 순수 GPU 임대비 all-in 학습비 추정
A. gpt-oss-120b급 5.1B 10T 8,900 0.2일 $0.7M $10M~$50M
B. 500B MoE 50B 30T 260,000 5.2일 $21.8M $100M~$300M
C. 1T MoE 100B 50T 868,000 17.4일 $72.9M $300M~$800M
D. 2T MoE 200B 80T 2,778,000 55.6일 $233.3M $700M~$1.5B+

강의 포인트:

  • 학습비는 크지만, 매일 반복되지는 않는다.
  • frontier 모델은 한 번의 최종 run만 있는 것이 아니라 수많은 실험과 post-training이 붙는다.
  • 그래서 모델 하나가 수억 달러에서 10억 달러급 프로젝트가 될 수 있다.

7. 추론비: 매일 나가는 비용

ChatGPT처럼 매일 수십억 request가 들어오면 추론비가 회사의 핵심 원가가 된다.

시나리오 가정:

  • ChatGPT weekly active users: 900M
  • 일일 request: 2.0B건
  • request당 평균 token: input 1,500 + output 500 = 2,000 tokens
  • 일일 처리 token: 4.0T tokens/day
  • H100 한 장의 실효 serving 처리량: 500 tokens/s
  • 평균 utilization: 40%

필요 H100 수:

4.0T tokens/day ÷ (500 tokens/s × 86,400s × 40%)
≈ 231,000 H100-equivalent GPUs
항목 값
일일 request 2.0B
request당 평균 token 2,000
일일 token 처리량 4.0T tokens
GPU당 실효 처리량 500 tokens/s
평균 utilization 40%
필요 GPU 약 231K H100-equivalent
총 HBM 약 18.5PB @80GB/GPU

이 숫자는 blended serving scenario다. 실제 OpenAI는 작은 모델, 큰 모델, reasoning 모델, image/audio/video 모델을 router로 나눠 쓰기 때문에 평균 비용은 요청 종류에 따라 달라진다.

8. 추론 원가 시나리오

H100 1장의 daily cost를 보수적으로 다음처럼 잡는다.

결론은 중요하다. OpenAI 같은 서비스는 “모델 하나를 학습하는 비용”보다 매일 추론으로 나가는 비용이 더 중요한 사업이 된다.

10. 수익성: gross margin과 회사 전체 손익은 다르다

OpenAI의 회사 전체 수익성은 공개되지 않았다. 따라서 unit economics를 나눠 봐야 한다.

항목 보수 중앙 공격
연 매출 $24B $24B $27B
추론비 $12B $7B $4B
serving gross profit $12B $17B $23B
serving gross margin 50% 71% 85%

여기서 다음 비용이 추가된다.

  • 모델 학습비
  • 연구개발 인력
  • 데이터센터 선투자
  • Microsoft/클라우드 revenue share
  • sales/enterprise support
  • safety/compliance
  • product 운영비
  • 무료 사용자 subsidization

그래서 회사 전체로는 매출이 커도 단기 순이익은 낮거나 적자일 수 있다. OpenAI는 회계상 이익보다 GPU/HBM capacity 확보와 사용자 lock-in을 우선하는 단계로 볼 수 있다.

11. 강의용 핵심 그림: 왜 HBM인가

OpenAI가 더 많은 사용자를 받고, 더 긴 context를 제공하고, 더 강한 reasoning model을 제공하려면 세 가지가 필요하다.

  1. 더 큰 HBM 용량: model weight와 KV cache를 올리기 위해
  2. 더 높은 HBM bandwidth: token 생성 속도를 유지하기 위해
  3. 더 낮은 $/token: 무료/저가 사용자를 감당하기 위해

12. 결론

  • OpenAI GPT-5 계열의 정확한 파라미터 수와 학습 GPU 수는 비공개다.
  • 공개된 gpt-oss-120b는 117B total, 5.1B active이고, 80GB GPU에 맞도록 설계됐다.
  • GPT-5급 frontier model은 시나리오상 500B~2T total parameter MoE 구조를 놓고 봐야 한다.
  • 1T MoE 모델은 BF16 weight만 2TB이고, FP8로 줄여도 1TB다.
  • ChatGPT가 900M weekly users, 수십억 request/day 규모라면 추론에는 수십만 H100-equivalent와 수십 PB HBM이 필요할 수 있다.
  • OpenAI의 매출은 $24B run-rate 수준까지 커졌지만, 추론비와 인프라 재투자 때문에 회사 전체 이익은 단기적으로 낮거나 적자일 수 있다.
  • 이 모든 구조가 DRAM/HBM 시장의 초호황과 직접 연결된다.
