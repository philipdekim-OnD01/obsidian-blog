오늘 배울 것

라즈베리파이에서 AI 모델을 돌릴 때 가장 자주 만나는 문제는 세 가지다.

모델 파일이 너무 크다.
추론 속도가 느리다.
메모리를 너무 많이 쓴다.

이 문제를 해결하는 대표 방법이 Quantization과 Pruning이다.

쉽게 말하면 다음과 같다.

Quantization: 숫자를 더 작게 표현한다.
Pruning: 덜 중요한 연결을 잘라낸다.

경량화의 목표는 단순히 파일 크기를 줄이는 것이 아니다. 온디바이스 AI에서는 다음 네 가지를 동시에 본다.

모델 크기
추론 속도
메모리 사용량
정확도 유지

파일 크기가 줄어도 FPS가 오르지 않을 수 있고, FPS가 올라도 정확도가 너무 떨어지면 쓸 수 없다. 그래서 경량화는 항상 측정과 함께 진행해야 한다.

왜 경량화가 필요한가?

PC나 GPU 서버는 메모리와 계산 능력이 크다. 하지만 라즈베리파이는 제한이 있다.

항목 PC/GPU 서버 Raspberry Pi
메모리 큼 작음
전력 충분함 제한적
냉각 팬/서버룸 작은 방열판 또는 팬
목적 최고 성능 적당한 성능과 빠른 반응

온디바이스 AI에서는 최고 정확도보다 작고 빠르고 안정적인 모델이 더 중요할 때가 많다.

Quantization 이해하기

딥러닝 모델의 가중치는 보통 32비트 실수다.

0.123456789
-1.827364
3.402113

Quantization은 이런 숫자를 더 작은 표현으로 바꾼다.

float32 → int8

비유하면, 아주 정밀한 자 대신 조금 덜 정밀하지만 가볍고 빠른 자를 쓰는 것이다.

딥러닝 모델은 수많은 곱셈과 덧셈을 한다. 숫자 하나하나를 float32로 계산하면 표현은 정밀하지만 메모리와 연산 부담이 커진다. int8은 표현 범위가 거칠지만 작고 빠르게 처리할 수 있다.

float32: 숫자 하나에 32비트
int8: 숫자 하나에 8비트

단, 모든 모델이 int8로 바꿔도 같은 정확도를 유지하는 것은 아니다. 가위바위보처럼 클래스가 단순한 모델은 잘 버티는 경우가 많지만, 미세한 차이를 구분해야 하는 모델은 정확도가 더 떨어질 수 있다.

Quantization의 장점

모델 파일 크기 감소
메모리 사용량 감소
추론 속도 향상 가능
전력 소모 감소 가능

예를 들어 32비트 가중치를 8비트로 바꾸면 이론적으로 가중치 저장 공간이 약 1/4로 줄어든다.

대표적인 Quantization 방법

방법 설명 난이도
Dynamic Range Quantization 가중치 중심으로 간단히 양자화 쉬움
Float16 Quantization float32를 float16으로 줄임 쉬움
Full Integer Quantization 입력/출력/중간 연산까지 int8로 양자화 중간
Quantization Aware Training 학습 중 양자화 오차를 미리 반영 어려움, 성능 좋음

처음 실습에서는 다음 순서가 좋다.

1. Dynamic Range Quantization으로 가장 쉬운 변환을 해 본다.
2. 정확도와 파일 크기를 비교한다.
3. 더 줄일 필요가 있으면 Full Integer Quantization을 시도한다.
4. 정확도 하락이 크면 Quantization Aware Training을 검토한다.

처음부터 어려운 방법으로 가면 학생들이 경량화의 효과를 체감하기 전에 설정 문제에 막힐 수 있다.

TFLite Dynamic Range Quantization 예시

import tensorflow as tf

converter = tf.lite.TFLiteConverter.from_saved_model("saved_model")
converter.optimizations = [tf.lite.Optimize.DEFAULT]
tflite_model = converter.convert()

with open("model_dynamic_range.tflite", "wb") as f:
    f.write(tflite_model)

가장 먼저 시도하기 좋은 방법이다. 코드가 짧고 적용이 쉽다.

변환 전후 파일 크기를 비교하는 코드는 다음과 같다.

from pathlib import Path

def print_size(path):
    size_mb = Path(path).stat().st_size / 1024 / 1024
    print(f"{path}: {size_mb:.2f} MB")

print_size("model_float32.tflite")
print_size("model_dynamic_range.tflite")

학생들이 숫자로 효과를 확인할 수 있게, 변환할 때마다 파일 크기를 기록한다.

Full Integer Quantization 개념

Full Integer Quantization은 더 강한 경량화다. 단, representative dataset이 필요하다.

representative dataset은 모델이 실제로 볼 입력 예시다.

def representative_dataset():
    for image in calibration_images:
        yield [image.astype("float32")]

왜 필요할까?

모델 안의 값이 어느 범위에서 움직이는지 알아야 int8로 잘 바꿀 수 있기 때문이다.

representative dataset은 테스트 데이터 전체일 필요는 없지만, 실제 입력을 대표해야 한다.

좋은 representative dataset:
라즈베리파이 카메라로 찍은 가위/바위/보 이미지가 골고루 들어 있음

나쁜 representative dataset:
밝은 조명에서 찍은 paper 이미지만 들어 있음

대표 데이터가 한쪽으로 치우치면 양자화 범위도 치우쳐서 실제 입력에서 정확도가 떨어질 수 있다.

Full Integer Quantization에서는 입력 dtype도 바뀔 수 있으므로 변환 후 반드시 다음을 확인한다.

interpreter = tflite.Interpreter(model_path="model_int8.tflite")
interpreter.allocate_tensors()
print(interpreter.get_input_details()[0]["dtype"])
print(interpreter.get_output_details()[0]["dtype"])

모델이 int8 입력을 기대하는데 float32 전처리를 그대로 쓰면 결과가 크게 흔들릴 수 있다.

Full Integer Quantization 변환 예시는 다음과 같다.

import tensorflow as tf
import numpy as np

calibration_images = np.load("calibration_images.npy")


def representative_dataset():
    for image in calibration_images[:100]:
        image = image.astype(np.float32)
        image = np.expand_dims(image, axis=0)
        yield [image]


converter = tf.lite.TFLiteConverter.from_saved_model("saved_model")
converter.optimizations = [tf.lite.Optimize.DEFAULT]
converter.representative_dataset = representative_dataset
converter.target_spec.supported_ops = [tf.lite.OpsSet.TFLITE_BUILTINS_INT8]
converter.inference_input_type = tf.uint8
converter.inference_output_type = tf.uint8

tflite_int8_model = converter.convert()

with open("model_full_int8.tflite", "wb") as f:
    f.write(tflite_int8_model)

calibration_images.npy는 실제 입력과 같은 크기의 이미지 배열이어야 한다.

예: (100, 224, 224, 3)

만약 모델이 64 x 64 입력을 기대한다면 calibration 이미지도 64 x 64로 맞춰야 한다.

Pruning 이해하기

Pruning은 모델 안에서 중요도가 낮은 가중치를 0으로 만드는 방법이다.

비유하면 나무 가지치기와 같다.

중요한 가지는 남기고
덜 중요한 가지는 잘라서
전체 나무를 가볍게 만든다.

딥러닝에서는 이런 식이다.

0.81  0.00  -0.73
0.00  0.00   0.42
0.15  0.00   0.00

0이 많아지면 sparse model이 된다.

다만 “0이 많다”와 “항상 빠르다”는 같은 말이 아니다. 일반적인 CPU나 런타임은 0이 많은 행렬을 자동으로 빠르게 처리하지 못할 수 있다.

Pruning의 직접 효과: 가중치 중 일부를 0으로 만든다.
실제 속도 향상: 런타임과 하드웨어가 sparse 연산을 잘 활용할 때 커진다.

그래서 라즈베리파이 수업에서는 pruning을 “압축과 실험 대상”으로 설명하고, 속도 향상은 반드시 측정해서 확인하게 해야 한다.

Pruning의 장점과 주의점

장점은 다음과 같다.

모델 압축 가능
일부 하드웨어에서 속도 향상 가능
불필요한 파라미터 제거

주의할 점도 있다.

무조건 빨라지는 것은 아니다.
하드웨어와 런타임이 sparse 연산을 잘 지원해야 한다.
너무 많이 pruning하면 정확도가 떨어진다.

Pruning 기본 절차

  1. 원래 모델을 학습한다.
  2. 중요도가 낮은 가중치를 점진적으로 0으로 만든다.
  3. pruning된 상태로 다시 학습해 정확도를 회복한다.
  4. strip pruning으로 학습용 마스크를 제거한다.
  5. TFLite로 변환한다.

개념 코드는 다음과 같다.

import tensorflow_model_optimization as tfmot

prune_low_magnitude = tfmot.sparsity.keras.prune_low_magnitude

pruned_model = prune_low_magnitude(
    model,
    pruning_schedule=tfmot.sparsity.keras.PolynomialDecay(
        initial_sparsity=0.0,
        final_sparsity=0.5,
        begin_step=0,
        end_step=1000,
    ),
)

final_sparsity=0.5는 가중치의 약 50%를 0으로 만들겠다는 뜻이다.

처음에는 30%, 50%, 70%처럼 여러 수준을 비교하면 좋다.

sparsity 예상 특징
30% 정확도 하락이 작을 가능성이 높음
50% 압축 효과와 정확도 사이의 균형 후보
70% 정확도 하락 위험이 커짐

Pruning 후에는 재학습이 중요하다. 가중치를 0으로 만든 직후에는 모델 성능이 떨어질 수 있으므로, pruning된 구조에 적응하도록 몇 epoch 더 학습해야 한다.

Pruning 학습 예시는 다음과 같다.

import tensorflow as tf
import tensorflow_model_optimization as tfmot

EPOCHS = 5
steps_per_epoch = len(train_ds)
end_step = steps_per_epoch * EPOCHS

pruning_params = {
    "pruning_schedule": tfmot.sparsity.keras.PolynomialDecay(
        initial_sparsity=0.0,
        final_sparsity=0.5,
        begin_step=0,
        end_step=end_step,
    )
}

pruned_model = tfmot.sparsity.keras.prune_low_magnitude(
    model,
    **pruning_params,
)

pruned_model.compile(
    optimizer="adam",
    loss="sparse_categorical_crossentropy",
    metrics=["accuracy"],
)

callbacks = [
    tfmot.sparsity.keras.UpdatePruningStep(),
]

pruned_model.fit(
    train_ds,
    validation_data=val_ds,
    epochs=EPOCHS,
    callbacks=callbacks,
)

stripped_model = tfmot.sparsity.keras.strip_pruning(pruned_model)
stripped_model.save("rps_pruned_saved_model")

Pruning을 적용할 때 UpdatePruningStep() 콜백을 넣지 않으면 pruning schedule이 제대로 진행되지 않는다.

Quantization과 Pruning을 같이 쓰기

두 방법은 함께 사용할 수 있다.

Pruning으로 필요 없는 가중치를 줄이고
Quantization으로 남은 숫자를 작게 표현한다.

추천 순서는 다음과 같다.

기본 모델 학습
→ Pruning 적용 및 재학습
→ strip pruning
→ Quantization
→ TFLite 변환
→ 라즈베리파이에서 속도/정확도 측정

여기서 strip pruning은 학습할 때 쓰던 pruning 관련 마스크를 제거하고 실제 배포용 모델로 정리하는 단계다. 이 단계를 빼면 모델 파일에 학습용 정보가 남아 배포가 깔끔하지 않을 수 있다.

두 방법을 같이 쓸 때는 결과가 항상 좋아진다고 가정하면 안 된다.

Pruning만 했을 때 정확도
Quantization만 했을 때 정확도
Pruning + Quantization 했을 때 정확도

세 결과를 따로 비교해야 한다. 어떤 모델은 quantization만 적용한 쪽이 더 실용적일 수 있다.

Pruning 후 TFLite로 변환하는 코드는 다음과 같다.

import tensorflow as tf

converter = tf.lite.TFLiteConverter.from_saved_model("rps_pruned_saved_model")
converter.optimizations = [tf.lite.Optimize.DEFAULT]

compressed_model = converter.convert()

with open("rps_pruned_quantized.tflite", "wb") as f:
    f.write(compressed_model)

Pruning 모델은 zip 압축했을 때 파일 크기 감소가 더 잘 보일 수 있다.

import zipfile
from pathlib import Path

def zip_model(model_path, zip_path):
    with zipfile.ZipFile(zip_path, "w", compression=zipfile.ZIP_DEFLATED) as zf:
        zf.write(model_path, arcname=Path(model_path).name)

zip_model("rps_pruned_quantized.tflite", "rps_pruned_quantized.zip")
print_size("rps_pruned_quantized.tflite")
print_size("rps_pruned_quantized.zip")

성능 측정은 반드시 장치에서 한다

Colab이나 PC에서 빠른 모델이 라즈베리파이에서도 빠르다는 보장은 없다.

반드시 라즈베리파이에서 다음을 직접 측정해야 한다.

측정 항목 의미
모델 파일 크기 저장 공간과 배포 편의성
평균 추론 시간 한 장 처리에 걸리는 시간
FPS 실시간 처리 가능성
정확도 경량화 후 성능 유지 여부
메모리 사용량 2GB 장치에서 안정성

추론 시간은 한 번만 재면 흔들림이 크다. 여러 번 실행해서 평균을 내야 한다.

import time

times = []
for _ in range(100):
    start = time.perf_counter()
    interpreter.invoke()
    end = time.perf_counter()
    times.append(end - start)

avg_ms = sum(times) / len(times) * 1000
fps = 1000 / avg_ms
print(f"avg: {avg_ms:.2f} ms, fps: {fps:.1f}")

실제로는 입력 텐서 설정까지 포함해서 측정하는 편이 더 현실적이다.

import time
import numpy as np
import tflite_runtime.interpreter as tflite

MODEL_PATH = "model_full_int8.tflite"

interpreter = tflite.Interpreter(model_path=MODEL_PATH)
interpreter.allocate_tensors()

input_details = interpreter.get_input_details()
output_details = interpreter.get_output_details()

input_shape = input_details[0]["shape"]
input_dtype = input_details[0]["dtype"]
dummy_input = np.zeros(input_shape, dtype=input_dtype)

# 처음 몇 번은 워밍업으로 제외한다.
for _ in range(10):
    interpreter.set_tensor(input_details[0]["index"], dummy_input)
    interpreter.invoke()

times = []
for _ in range(100):
    start = time.perf_counter()
    interpreter.set_tensor(input_details[0]["index"], dummy_input)
    interpreter.invoke()
    output = interpreter.get_tensor(output_details[0]["index"])
    end = time.perf_counter()
    times.append(end - start)

avg_ms = sum(times) / len(times) * 1000
print("output shape:", output.shape)
print(f"average inference time: {avg_ms:.2f} ms")
print(f"estimated FPS: {1000 / avg_ms:.1f}")

이 코드는 카메라 없이 모델 속도만 재는 코드다. 카메라까지 포함한 FPS가 낮다면 모델 추론 시간과 전체 루프 시간을 따로 비교해야 한다.

loop_start = time.perf_counter()
ret, frame = cap.read()
label, confidence = predict(frame)
loop_end = time.perf_counter()

loop_ms = (loop_end - loop_start) * 1000
print(f"camera + preprocess + inference: {loop_ms:.2f} ms")

카메라 입력, 전처리, 모델 추론, 화면 출력 중 어디가 느린지도 분리해서 봐야 한다.

전체 FPS가 낮다
→ 모델이 느린가?
→ resize가 느린가?
→ 화면 출력이 느린가?
→ 카메라 프레임 읽기가 느린가?

이렇게 나누어 봐야 경량화가 필요한지, 코드 최적화가 필요한지 판단할 수 있다.

실습 기록표

모델 크기 정확도 평균 추론 시간 FPS 메모
기본 TFLite 기준 모델
Dynamic Range Quantization 가장 쉬운 양자화
Full INT8 Quantization representative dataset 필요
Pruning 50% 정확도 하락 확인
Pruning + Quantization 최종 후보

기록표에는 숫자뿐 아니라 판단도 적어야 한다.

파일 크기는 줄었지만 FPS 변화가 거의 없음
정확도는 1%p 떨어졌지만 FPS가 크게 올라 실시간성이 좋아짐
70% pruning은 너무 많이 줄여서 가위와 보를 자주 헷갈림

이런 해석이 있어야 단순 실습이 아니라 엔지니어링 판단이 된다.

실습 과제

  1. 기본 TFLite 모델의 크기와 FPS를 측정한다.
  2. Dynamic Range Quantization을 적용한다.
  3. Full INT8 Quantization을 적용한다.
  4. Pruning 30%, 50%, 70%를 비교한다.
  5. 가장 좋은 모델을 골라 이유를 설명한다.

최종 제출물에는 다음 결론을 포함한다.

  1. 라즈베리파이에서 가장 실용적인 모델
  2. 선택 이유: 크기, FPS, 정확도 중 무엇을 우선했는가
  3. 정확도 하락이 있었다면 허용 가능한 수준인가
  4. 다음에 더 개선한다면 데이터, 모델, 경량화 중 무엇을 바꿀 것인가

오늘의 핵심 정리

Quantization은 숫자의 표현을 줄이는 방법이다.
Pruning은 덜 중요한 가중치를 제거하는 방법이다.
경량화는 정확도, 속도, 크기 사이의 균형 문제다.
최종 평가는 반드시 라즈베리파이에서 해야 한다.
