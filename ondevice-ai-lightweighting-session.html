사례: 70B LLM은 왜 HBM을 많이 쓰는가?

LLM에서 70B 모델이라고 하면 보통 파라미터, 즉 weight 숫자가 약 700억 개 있다는 뜻입니다.

70B = 70 billion parameters
    = 약 700억 개의 weight 숫자

모델의 weight 메모리는 아주 단순하게는 아래처럼 계산할 수 있습니다.

모델 weight 메모리 ≈ 파라미터 개수 × 숫자 1개의 저장 크기

float16 또는 bfloat16은 숫자 하나를 2 byte로 저장합니다. 따라서 70B 모델을 FP16으로 저장하면 weight만 약 140GB가 필요합니다.

70B FP16 모델
= 700억 개 × 2 byte
= 1400억 byte
= 약 140GB

즉, 70B 모델을 FP16으로 GPU에 올리려면 weight만으로도 약 140GB의 HBM이 필요합니다.

숫자 표현에 따른 70B 모델 weight 크기

표현 방식 숫자 1개 크기 70B 모델 weight 크기 FP16 대비
FP32 4 byte 약 280GB 2배
FP16 / BF16 2 byte 약 140GB 기준
INT8 1 byte 약 70GB 절반
INT4 0.5 byte 약 35GB 1/4

이 표에서 볼 수 있듯이, FP16 -> INT8로 줄이면 weight 메모리는 약 절반이 됩니다. FP16 -> INT4로 줄이면 weight 메모리는 약 1/4이 됩니다.

70B FP16: 약 140GB
70B INT8: 약 70GB
70B INT4: 약 35GB

왜 추론에서도 실제 메모리는 더 필요한가?

위 계산은 오직 weight만 계산한 것입니다. 실제 추론에는 추가 메모리가 필요합니다.

실제 추론 메모리
= 모델 weight
+ KV cache
+ activation
+ runtime overhead
+ CUDA/kernel workspace

특히 LLM은 긴 문맥을 처리할수록 KV cache가 커집니다.

짧은 질문:
weight 메모리 비중이 큼

긴 대화 / 긴 문서:
weight + KV cache 메모리가 모두 중요

그래서 LLM 온디바이스 추론에서는 weight quantization뿐 아니라 KV cache compression도 중요해집니다.

학습에서는 왜 더 많은 메모리가 필요한가?

추론은 주로 학습된 weight를 읽어서 답을 생성합니다. 하지만 학습은 weight를 고쳐야 하므로 더 많은 정보를 저장해야 합니다.

학습 메모리
= weight
+ gradient
+ optimizer state
+ activation
+ batch data

그래서 경량화는 추론에서는 GPU와 HBM 요구량을 직접 줄여 줍니다. 반면 학습에서는 전체 사전학습보다 fine-tuning에서 효과가 더 큽니다.

핵심

70B는 weight 숫자가 약 700억 개라는 뜻이다.
FP16은 숫자 하나가 2 byte이므로 weight만 약 140GB가 필요하다.
INT8로 줄이면 약 70GB, INT4로 줄이면 약 35GB까지 줄어든다.
경량화는 추론에서는 GPU/HBM 요구량을 직접 줄인다.
학습에서는 전체 학습보다 LoRA, QLoRA 같은 fine-tuning에서 효과가 크다.

보조 사례: 왜 Transformer가 LLM의 기본 구조가 되었는가?

LLM을 이해하려면 먼저 Transformer가 왜 등장했는지 알아야 합니다. Transformer 이전에는 RNN, LSTM, GRU 같은 순환 신경망이 문장 처리에 많이 쓰였습니다.

RNN은 문장을 왼쪽에서 오른쪽으로 하나씩 읽습니다.

나는 -> 오늘 -> 학교에 -> 갔다

이 방식은 자연스럽지만 긴 문장을 처리할 때 문제가 있습니다.

RNN 계열의 한계 의미
기울기 소실/폭주 문장이 길어질수록 앞쪽 정보가 뒤로 잘 전달되지 않음
긴 문맥 처리 어려움 멀리 떨어진 단어 관계를 잡기 어려움
순차 계산 단어를 하나씩 처리하므로 GPU 병렬처리에 불리함
정보 압축 부담 hidden state 하나에 이전 문맥을 계속 압축해야 함

LSTM과 GRU는 이 문제를 줄였지만 완전히 해결하지는 못했습니다.

Transformer의 핵심 발상은 다음과 같습니다.

단어를 하나씩 순서대로 보지 말고,
문장 안의 모든 단어가 서로를 직접 보게 하자.

이것이 self-attention입니다.

예를 들어 다음 문장을 봅니다.

The animal did not cross the street because it was too tired.

여기서 it이 무엇을 가리키는지 알려면 앞의 animal, street와의 관계를 봐야 합니다. RNN은 정보를 순서대로 전달하지만, Transformer는 it이 문장 안의 다른 단어들을 직접 참고할 수 있습니다.

RNN:
앞 단어의 정보를 다음 단어로 계속 전달

Transformer:
각 단어가 문장 안의 다른 모든 단어를 직접 참고

Attention Is All You Need 논문은 2017년에 Google Brain/Google Research 팀이 발표했습니다. 저자는 Ashish Vaswani, Noam Shazeer, Niki Parmar, Jakob Uszkoreit, Llion Jones, Aidan Gomez, Lukasz Kaiser, Illia Polosukhin입니다.

이 논문의 중요한 전환점은 attention을 RNN의 보조 장치로 쓰는 것이 아니라, 모델의 중심 구조로 만든 것입니다.

이전 생각:
RNN이 기본이고 attention은 보조 도구

Transformer의 생각:
RNN 없이 attention만으로 sequence model을 만들 수 있다

그래서 논문 제목이 Attention Is All You Need입니다.

RNN과 Transformer 비교

구분 RNN/LSTM Transformer
처리 방식 단어를 순서대로 처리 모든 단어 관계를 한 번에 계산
병렬화 어려움 GPU 병렬화에 유리
긴 문맥 앞 정보가 약해질 수 있음 멀리 떨어진 단어도 직접 연결
핵심 저장 방식 hidden state에 문맥 압축 attention으로 필요한 정보 조회
확장성 대규모 학습에 상대적으로 불리 모델과 데이터가 커질수록 성능 확장에 유리

쉽게 말하면 다음과 같습니다.

RNN은 문장을 줄 서서 전달하는 방식이고,
Transformer는 모든 단어가 회의실에 같이 앉아 서로를 보는 방식입니다.

경량화와의 연결

Transformer는 GPU/HBM의 대규모 행렬 연산 능력을 잘 활용합니다. 모든 파라미터를 메모리에 올려 두고, attention과 feed-forward layer의 큰 행렬곱을 반복하면서 높은 성능을 냅니다.

하지만 이 장점은 동시에 비용을 만듭니다.

모델이 커질수록:
weight 메모리 증가
KV cache 증가
GPU/HBM 요구량 증가
전력과 발열 증가

그래서 LLM 시대의 경량화는 단순히 모델 파일을 작게 만드는 문제가 아닙니다.

Transformer의 장점:
병렬 연산, 긴 문맥 이해, 대규모 확장성

Transformer의 비용:
큰 weight, 큰 KV cache, 많은 HBM, 높은 전력

LLM 경량화의 목표:
Transformer의 장점은 유지하면서 실행 비용을 줄이는 것

핵심

Transformer는 RNN의 기울기 소실, 긴 문맥 처리, 병렬화 한계를 넘기 위해 나왔다.
핵심 아이디어는 단어를 순서대로 기억하지 말고, 모든 단어가 서로를 직접 보게 하는 self-attention이다.
LLM 경량화는 Transformer의 강력한 성능은 최대한 유지하면서,
weight와 KV cache가 요구하는 GPU/HBM 비용을 줄이기 위한 기술이다.

네 가지 기법을 한 문장으로

Pruning

모델 안의 “별로 중요하지 않은 연결”을 0으로 만듭니다.

많은 연결 -> 중요한 연결만 남김

Clustering

비슷한 가중치 숫자들을 같은 대표 숫자로 바꿉니다.

수많은 숫자 종류 -> 몇 개의 대표 숫자

PTQ

학습이 끝난 모델을 TFLite로 바꾸면서 숫자 크기를 줄입니다.

float32 -> float16 또는 int8

QAT

숫자가 작게 바뀌어도 잘 맞히도록 학습 중에 미리 연습시킵니다.

양자화될 상황을 미리 경험하며 학습

가장 헷갈리는 부분

Pruning과 Clustering 차이

Pruning은 값을 0으로 만드는 것입니다.

0.52, 0.01, -0.33 -> 0.52, 0, -0.33

Clustering은 비슷한 값을 같은 값으로 묶는 것입니다.

0.52, 0.50, 0.49 -> 0.50, 0.50, 0.50

PTQ와 QAT 차이

PTQ는 시험이 끝난 뒤 답안지를 줄여 쓰는 느낌입니다. 쉽지만 글씨가 뭉개져서 의미가 조금 달라질 수 있습니다.

QAT는 처음부터 작은 칸에 답을 쓰는 연습을 하며 공부하는 느낌입니다. 더 번거롭지만 최종 결과가 안정적일 수 있습니다.

  2. 그 다음 “실습에서 확인할 것”을 봅니다.

온디바이스 AI와 경량화 개요

먼저 한 문장으로

온디바이스 AI 경량화는 “큰 AI 모델을 작은 기기에서도 돌아가도록 가볍게 정리하는 작업”입니다.

온디바이스 AI란?

온디바이스 AI는 서버로 데이터를 보내지 않고, 스마트폰, 라즈베리파이, 마이크로컨트롤러, 카메라 같은 기기 안에서 AI 모델을 직접 실행하는 방식입니다.

장점은 다음과 같습니다.

  • 인터넷이 불안정해도 동작한다.
  • 서버 왕복 시간이 없어 반응이 빠르다.
  • 이미지, 음성 같은 민감한 데이터를 외부로 보내지 않아도 된다.
  • 서버 비용을 줄일 수 있다.

하지만 기기 안에서 실행하려면 제약이 있습니다.

  • 메모리가 작다.
  • CPU/GPU/NPU 성능이 서버보다 낮다.
  • 배터리를 아껴야 한다.
  • 모델 파일이 너무 크면 배포와 로딩이 느리다.

그래서 경량화가 필요합니다.

쉬운 비유

서버는 넓은 연구실이고, 온디바이스 기기는 작은 책상입니다. 연구실에서는 큰 장비를 마음껏 펼칠 수 있지만, 작은 책상에서는 꼭 필요한 도구만 올려야 합니다.

AI 모델도 같습니다.

  • 서버에서는 큰 모델을 돌릴 수 있습니다.
  • 라즈베리파이에서는 큰 모델이 느리거나 메모리 부족으로 안 될 수 있습니다.
  • 그래서 모델을 작게 줄이고, 계산을 단순하게 만들고, 실행 파일을 가볍게 해야 합니다.

경량화가 줄이는 것

줄이는 대상 의미 대표 기법
모델 파일 크기 저장 공간, 다운로드 크기 Quantization, Clustering
메모리 사용량 실행 중 RAM 사용 Quantization, Pruning
연산량 추론 시간, 전력 사용 Pruning, Quantization
중복된 표현 비슷한 가중치 반복 Clustering

이 예제에서 배우는 4가지 축

1. Pruning

중요도가 낮은 가중치를 0으로 만드는 방법입니다. 0이 많아지면 sparse model이 됩니다.

핵심 질문:

  • 어느 가중치를 0으로 만들 것인가?
  • 0을 많이 만들었을 때 정확도는 얼마나 떨어지는가?
  • 압축했을 때 실제 파일 크기가 줄어드는가?

관련 노트: Pruning: 가지치기

2. Clustering

서로 비슷한 가중치들을 몇 개의 대표값으로 묶는 방법입니다. 예를 들어 수많은 가중치를 16개의 대표값만 사용하도록 제한합니다.

핵심 질문:

  • 대표값 개수를 몇 개로 둘 것인가?
  • 같은 대표값을 공유하면 압축이 얼마나 잘 되는가?
  • 일부 레이어에만 적용하면 정확도 손실을 줄일 수 있는가?

관련 노트: Clustering: 가중치 군집화

3. PTQ

이미 학습된 모델을 TFLite로 변환하면서 양자화하는 방법입니다. 학습을 다시 하지 않거나 거의 하지 않기 때문에 적용이 쉽습니다.

핵심 질문:

  • Dynamic Range, INT8, FLOAT16은 무엇이 다른가?
  • 대표 데이터셋이 왜 필요한가?
  • 입력과 출력 dtype이 바뀌면 추론 코드는 어떻게 달라지는가?

관련 노트: PTQ: 학습 후 양자화

4. QAT

학습 중에 양자화될 상황을 미리 흉내 내는 방법입니다. 양자화 후 정확도 하락을 줄이는 데 유리합니다.

핵심 질문:

  • PTQ보다 번거로운데 왜 쓰는가?
  • 학습 그래프에 quantize wrapper가 생기는 이유는 무엇인가?
  • 최종 .tflite 변환은 어떻게 하는가?

1. 라이브러리 설치

주요 라이브러리는 TensorFlow와 TensorFlow Model Optimization Toolkit입니다.

!pip install tensorflow-model-optimization
import tensorflow as tf
import tensorflow_model_optimization as tfmot

tfmot는 pruning, clustering, QAT를 제공하는 핵심 도구입니다.

초보자식으로 말하면 tfmot는 TensorFlow 모델을 가볍게 만드는 도구 상자입니다.

2. 데이터 로딩

MNIST는 손글씨 숫자 이미지 데이터셋입니다.

(train_images, train_labels), (test_images, test_labels) = keras.datasets.mnist.load_data()
train_images = train_images / 255.0
test_images = test_images / 255.0

이미지 픽셀은 원래 0부터 255 사이의 정수입니다. 이를 0부터 1 사이 값으로 바꾸면 모델이 학습하기 쉽습니다.

3. Baseline 모델

Baseline은 경량화 전 기준 모델입니다. 실험에서 기준 모델이 없으면 경량화가 좋아졌는지 나빠졌는지 판단할 수 없습니다.

쉽게 말해 baseline은 “원래 성적표”입니다. 경량화를 한 뒤에는 항상 원래 성적표와 비교해야 합니다.

확인해야 할 값:

  • baseline test accuracy
  • baseline model file size
  • baseline TFLite file size

4. Fine-tuning

Pruning, Clustering, QAT는 모델 구조나 학습 조건을 바꿉니다. 그래서 적용 직후 바로 끝내지 않고 짧게 다시 학습합니다. 이 과정을 fine-tuning이라고 합니다.

model.fit(train_images, train_labels, epochs=1 또는 2)

초보자는 fine-tuning을 “경량화로 흔들린 모델을 다시 적응시키는 단계”로 이해하면 됩니다.

비유하면 책상 배치를 바꾼 뒤, 다시 익숙해지도록 몇 번 연습하는 과정입니다.

5. TFLite 변환

온디바이스 실행을 위해 Keras 모델을 TFLite 모델로 바꿉니다.

converter = tf.lite.TFLiteConverter.from_keras_model(model)
tflite_model = converter.convert()

양자화를 적용할 때는 다음 옵션이 자주 등장합니다.

converter.optimizations = [tf.lite.Optimize.DEFAULT]

Pruning: 가지치기

대응 노트북:

  • EX_01_Pruning_Whole_Model.ipynb
  • EX_02_Pruning_Some_Layer.ipynb

핵심 개념

Pruning은 모델의 가중치 중 중요도가 낮은 값을 0으로 만드는 기법입니다.

아주 쉽게 말하면 “AI 모델 안의 약한 연결을 끊는 것”입니다.

쉬운 비유

신경망을 도로망이라고 생각해 봅니다. 도시 안에 도로가 너무 많으면 관리가 어렵습니다. 그런데 실제로는 거의 차가 다니지 않는 작은 길도 있습니다. Pruning은 이런 거의 안 쓰는 길을 막고, 중요한 길만 남기는 작업입니다.

복잡한 도로망
-> 거의 안 쓰는 길 제거
-> 더 단순한 도로망

모델에서는 “길”이 가중치이고, “길을 막는다”는 것이 가중치를 0으로 만든다는 뜻입니다.

예를 들어 가중치가 다음과 같다고 합시다.

[0.82, 0.01, -0.44, 0.002, 0.31]

작은 값들을 0으로 만들면 다음처럼 됩니다.

[0.82, 0, -0.44, 0, 0.31]

0이 많아진 모델은 sparse model입니다.

왜 경량화가 되는가?

0이 많아지면 압축이 잘 됩니다. 다만 일반 .h5 파일만 보면 크기가 확 줄지 않을 수 있습니다. 실제 이득은 압축 저장, sparse 연산 지원, TFLite 변환과 함께 볼 때 더 의미가 있습니다.

EX_01: 전체 모델 Pruning

전체 모델에 pruning wrapper를 씌웁니다.

pruning_params = {
    "pruning_schedule": tfmot.sparsity.keras.PolynomialDecay(
        initial_sparsity=0.50,
        final_sparsity=0.80,
        begin_step=0,
        end_step=end_step
    )
}

model_for_pruning = tfmot.sparsity.keras.prune_low_magnitude(
    model,
    **pruning_params
)

읽는 법:

  • initial_sparsity=0.50: 처음부터 가중치의 약 50%를 0으로 만들겠다는 뜻입니다.
  • final_sparsity=0.80: 최종적으로 약 80%까지 0으로 만들겠다는 뜻입니다.
  • PolynomialDecay: 한 번에 확 자르지 않고 학습 step에 따라 점진적으로 0을 늘립니다.

꼭 필요한 callback

callbacks = [
    tfmot.sparsity.keras.UpdatePruningStep(),
]

Pruning schedule은 현재 학습 step을 알아야 동작합니다. 이 callback이 없으면 pruning이 의도대로 진행되지 않습니다.

학습 후 wrapper 제거

final_model = tfmot.sparsity.keras.strip_pruning(model_for_pruning)

prune_low_magnitude를 적용하면 모델에 pruning용 부가 정보가 붙습니다. 배포 전에는 strip_pruning으로 학습용 wrapper를 제거합니다.

EX_02: 일부 레이어만 Pruning

모든 레이어에 pruning을 걸면 정확도가 더 흔들릴 수 있습니다. 그래서 Dense 레이어처럼 파라미터가 많은 일부 레이어에만 적용할 수 있습니다.

def apply_pruning_to_dense(layer):
    if isinstance(layer, tf.keras.layers.Dense):
        return tfmot.sparsity.keras.prune_low_magnitude(layer, **pruning_params)
    return layer

model_for_pruning = keras.models.clone_model(
    model,
    clone_function=apply_pruning_to_dense,
)

읽는 법:

  • clone_model: 기존 모델을 복사하면서 레이어별로 수정할 수 있습니다.
  • isinstance(layer, Dense): Dense 레이어만 pruning 대상입니다.
  • Conv2D 같은 레이어는 그대로 둡니다.

실습에서 확인할 것

  • baseline accuracy와 pruned accuracy가 얼마나 차이 나는가?
  • model.summary()에서 pruning wrapper가 붙었는가?
  • strip_pruning 후 최종 모델 구조가 단순해졌는가?
  • 압축 파일 크기가 줄었는가?

초보자가 자주 하는 오해

오해 실제 의미
0으로 만들면 무조건 빨라진다 실행 엔진이 sparse 연산을 잘 지원해야 속도 이득이 커집니다.
pruning을 많이 할수록 항상 좋다 너무 많이 자르면 정확도가 떨어집니다.
pruning 후 바로 배포하면 된다 보통 fine-tuning과 strip_pruning이 필요합니다.

초보자 포인트

Pruning은 “작은 가중치를 0으로 만드는 것”입니다. 정확도 하락을 줄이려면 pruning 후 fine-tuning을 해야 합니다.

Clustering: 가중치 군집화

대응 노트북:

  • EX_03_Clustering_Whole_Model.ipynb
  • EX_04_Clustering_Some_Layer.ipynb

핵심 개념

Clustering은 많은 가중치를 몇 개의 대표값으로 묶는 방법입니다.

아주 쉽게 말하면 “비슷비슷한 숫자는 같은 숫자로 통일하는 것”입니다.

쉬운 비유

색연필이 1,000개 있는데 색이 아주 조금씩만 다르다고 해 봅니다. 실제로 그림을 그릴 때는 빨강 계열 16개, 파랑 계열 16개 정도만 있어도 충분할 수 있습니다.

Clustering은 모델 안의 수많은 숫자를 이런 대표 색연필 몇 개로 정리하는 작업입니다.

수많은 미세한 색 차이
-> 대표 색 몇 개로 묶기
-> 저장하기 쉬워짐

예를 들어 가중치가 10,000개 있어도 실제 값 종류를 16개만 쓰도록 제한할 수 있습니다.

원래 가중치: 0.123, 0.127, 0.131, -0.45, -0.47, ...
군집화 후:   0.13,  0.13,  0.13,  -0.46, -0.46, ...

왜 압축이 되는가?

모든 가중치 값을 각각 저장하는 대신, 대표값 목록과 각 가중치가 어떤 대표값을 쓰는지만 저장할 수 있습니다. 그래서 zip 같은 압축에서 효과가 커집니다.

EX_03: 전체 모델 Clustering

clustering_params = {
    "number_of_clusters": 16,
    "cluster_centroids_init": tfmot.clustering.keras.CentroidInitialization.LINEAR
}

clustered_model = tfmot.clustering.keras.cluster_weights(
    model,
    **clustering_params
)

읽는 법:

  • number_of_clusters=16: 가중치 대표값을 16개로 제한합니다.
  • CentroidInitialization.LINEAR: 대표값 초기 위치를 선형적으로 배치합니다.
  • cluster_weights: 모델에 clustering wrapper를 적용합니다.

EX_04: 일부 레이어만 Clustering

레이어마다 대표값 개수를 다르게 줄 수 있습니다.

dict_clustering_params = {
    "conv2d_1": {
        "number_of_clusters": 8,
        "cluster_centroids_init": tfmot.clustering.keras.CentroidInitialization.KMEANS_PLUS_PLUS
    },
    "dense": {
        "number_of_clusters": 16,
        "cluster_centroids_init": tfmot.clustering.keras.CentroidInitialization.KMEANS_PLUS_PLUS
    }
}

읽는 법:

  • conv2d_1은 8개 대표값만 사용합니다.
  • dense는 16개 대표값을 사용합니다.
  • KMEANS_PLUS_PLUS는 데이터 분포를 고려해 대표값 초기 위치를 잡는 방식입니다.

학습 후 wrapper 제거

final_model = tfmot.clustering.keras.strip_clustering(clustered_model)

Pruning과 마찬가지로, clustering도 학습용 wrapper를 제거한 뒤 저장하거나 변환하는 것이 좋습니다.

PTQ: 학습 후 양자화

대응 노트북:

  • EX_05_PTQ_Dynamic_Range.ipynb
  • EX_06_PTQ_INT8.ipynb
  • EX_07_PTQ_FLOAT16.ipynb

핵심 개념

PTQ는 Post-Training Quantization의 약자입니다. 이미 학습된 모델을 다시 오래 학습하지 않고, TFLite 변환 단계에서 더 작은 숫자 표현으로 바꿉니다.

딥러닝 모델은 보통 float32를 많이 씁니다. float32는 숫자 하나에 32비트를 씁니다. 이를 float16 또는 int8로 바꾸면 모델 크기와 메모리 사용량을 줄일 수 있습니다.

쉬운 비유

정밀한 저울이 있다고 생각해 봅니다.

  • float32: 0.123456789g처럼 매우 자세히 기록합니다.
  • float16: 0.1235g 정도로 덜 자세히 기록합니다.
  • int8: 정해진 눈금 256칸 중 하나로 기록합니다.

기록은 덜 정밀해지지만, 종이에 적는 양과 계산 부담은 줄어듭니다. 양자화는 이 생각과 비슷합니다.

숫자 크기 감각

float32: 숫자 하나를 32비트로 저장
float16: 숫자 하나를 16비트로 저장
int8:    숫자 하나를 8비트로 저장

단순히 저장 크기만 보면 int8은 float32보다 4배 작습니다. 실제 모델 전체 크기는 레이어 구성과 변환 방식에 따라 달라집니다.

공통 변환 코드

converter = tf.lite.TFLiteConverter.from_keras_model(model)
converter.optimizations = [tf.lite.Optimize.DEFAULT]
tflite_model = converter.convert()

Optimize.DEFAULT는 TensorFlow Lite가 가능한 기본 최적화를 적용하도록 지시합니다.

EX_05: Dynamic Range Quantization

Dynamic Range 방식은 가장 적용하기 쉽습니다.

특징:

  • 별도 대표 데이터셋 없이 적용 가능
  • 주로 weight를 양자화
  • 입력과 출력은 float인 경우가 많아 기존 추론 코드와 비슷함
  • 가장 간단하지만 INT8 full quantization보다 최적화 폭이 작을 수 있음

초보자용 이해:

Dynamic Range PTQ는 “일단 쉽게 모델 크기를 줄여보는 첫 번째 양자화”입니다.

EX_06: INT8 Quantization

INT8은 가중치와 활성값을 8비트 정수 중심으로 바꾸는 방식입니다.

def representative_data_gen():
    for input_value in tf.data.Dataset.from_tensor_slices(train_images).batch(1).take(100):
        yield [input_value]

converter.target_spec.supported_ops = [tf.lite.OpsSet.TFLITE_BUILTINS_INT8]
converter.inference_input_type = tf.int8
converter.inference_output_type = tf.int8
converter.representative_dataset = representative_data_gen

읽는 법:

  • representative_dataset: 실제 입력과 비슷한 샘플입니다.
  • supported_ops: INT8 연산만 사용하도록 요청합니다.
  • inference_input_type=tf.int8: 모델 입력도 int8입니다.
  • inference_output_type=tf.int8: 모델 출력도 int8입니다.

대표 데이터셋이 필요한 이유

float 값을 int8로 바꾸려면 “어떤 float 범위를 int8 범위에 대응시킬지” 정해야 합니다. 이 범위를 잡는 과정을 calibration이라고 생각하면 됩니다. 대표 데이터셋은 calibration에 쓰입니다.

INT8 추론에서 주의할 점

모델 입력이 int8이면 이미지를 그냥 float로 넣으면 안 됩니다. TFLite interpreter의 input_details에서 scale과 zero point를 확인해 입력을 변환해야 합니다.

개념적으로는 다음과 같습니다.

real_value = scale * (int8_value - zero_point)

또는 반대로 입력 float를 int8로 넣을 때는 scale과 zero point를 이용해 변환합니다.

EX_07: FLOAT16 Quantization

converter.target_spec.supported_types = [tf.float16]

특징:

  • float32 weight를 float16으로 줄임
  • GPU가 float16에 강한 환경에서 유리할 수 있음
  • 입력과 출력은 float32로 유지되는 경우가 많아 사용이 편함
  • INT8보다 정확도 손실이 작을 가능성이 있음

세 방식 비교

방식 적용 난이도 대표 데이터셋 입력/출력 dtype 특징
Dynamic Range 낮음 보통 불필요 주로 float 가장 간단
INT8 중간 필요 int8 가능 가장 강한 경량화 후보
FLOAT16 낮음 불필요 주로 float 정확도 보존이 쉬운 편

PTQ 세 가지를 아주 쉽게 고르기

원하는 것 먼저 볼 방식
가장 쉽게 해보고 싶다 Dynamic Range
크기와 속도를 강하게 줄이고 싶다 INT8
정확도 손실을 적게 하면서 크기를 줄이고 싶다 FLOAT16

INT8이 어려운 이유

INT8은 숫자를 아주 작은 칸에 넣습니다. 그래서 입력 이미지도 그 칸에 맞춰 넣어야 합니다.

원래 이미지 값: 0.0부터 1.0 사이 float
INT8 모델 입력: -128부터 127 사이 정수

따라서 INT8 모델에서는 input_details에 있는 scale과 zero point를 보고 입력을 맞춰야 합니다.

  • input_details, output_details의 dtype이 무엇인가?
  • INT8 모델에서 scale과 zero point가 출력되는가?
  • baseline accuracy와 PTQ 모델 accuracy 차이가 허용 가능한가?

초보자 포인트

PTQ는 “학습이 끝난 모델을 변환하면서 숫자 표현을 작게 바꾸는 것”입니다. 쉽지만, INT8처럼 강하게 줄일수록 입력 처리와 정확도 확인이 중요해집니다.

QAT: 양자화 인지 학습

대응 노트북:

  • EX_08_Quantization_Aware_Training.ipynb

핵심 개념

QAT는 Quantization Aware Training의 약자입니다. 학습할 때부터 “나중에 양자화될 것”을 모델이 미리 경험하게 만드는 방법입니다.

PTQ는 학습이 끝난 뒤 숫자 표현을 바꿉니다. 반면 QAT는 학습 과정에서 양자화 오차를 흉내 냅니다.

쉬운 비유

PTQ는 넓은 공책에 글씨를 다 쓴 다음, 마지막에 작은 종이에 줄여 쓰는 것과 비슷합니다. 글자가 뭉개질 수 있습니다.

QAT는 처음부터 작은 칸 공책에 맞춰 쓰는 연습을 하며 공부하는 것과 비슷합니다. 시간이 더 걸리지만 작은 칸에 옮겼을 때 결과가 더 안정적입니다.

왜 QAT가 필요한가?

양자화는 숫자를 거칠게 표현합니다.

float32: 0.123456789
int8로 표현 후 복원: 0.125 정도

이 작은 차이가 레이어를 지나며 누적되면 정확도가 떨어질 수 있습니다. QAT는 모델이 이런 오차에 익숙해지도록 학습합니다.

QAT 모델 만들기

q_aware_model = tfmot.quantization.keras.quantize_model(model)

읽는 법:

  • 기존 Keras 모델을 QAT용 모델로 감쌉니다.
  • 내부에 quantize 관련 layer/wrapper가 추가됩니다.
  • 이 상태에서 다시 fine-tuning합니다.

Fine-tuning

q_aware_model.compile(
    optimizer="adam",
    loss=keras.losses.SparseCategoricalCrossentropy(),
    metrics=["accuracy"]
)

q_aware_model.fit(
    train_images,
    train_labels,
    batch_size=500,
    epochs=1,
    validation_split=0.1
)

초보자는 이 단계를 “양자화 환경에 적응시키는 추가 학습”으로 이해하면 됩니다.

TFLite 변환

converter = tf.lite.TFLiteConverter.from_keras_model(q_aware_model)
converter.optimizations = [tf.lite.Optimize.DEFAULT]
tflite_qat_model = converter.convert()

QAT도 최종 배포를 위해서는 TFLite 변환이 필요합니다.

PTQ와 QAT 비교

구분 PTQ QAT
적용 시점 학습 후 변환 단계 학습 중
구현 난이도 쉬움 더 복잡함
추가 학습 거의 없음 필요
정확도 보존 모델에 따라 다름 보통 더 유리
사용 상황 빠른 실험, 간단 배포 정확도가 중요한 배포

PTQ와 QAT를 그림으로 비교

PTQ
학습 완료 -> 양자화 -> 정확도 확인

QAT
학습된 모델 -> 양자화 상황을 흉내 내며 추가 학습 -> 양자화 -> 정확도 확인

핵심 차이는 “양자화 오차를 학습 중에 미리 경험했는가”입니다.

언제 QAT까지 해야 하나?

처음부터 QAT를 할 필요는 없습니다. 보통은 다음 순서가 실용적입니다.

1. baseline 모델 만들기
2. PTQ 적용해 보기
3. 정확도가 충분하면 PTQ로 끝
4. 정확도가 많이 떨어지면 QAT 시도

실습에서 확인할 것

  • q_aware_model.summary()에서 quantize 관련 구조가 보이는가?
  • QAT fine-tuning 후 accuracy가 유지되는가?
  • 변환된 mnist_qat_model.tflite 크기가 baseline보다 줄었는가?
  • TFLite interpreter에서 입력과 출력 dtype이 무엇인가?

초보자 포인트

QAT는 “양자화될 미래를 알고 학습하는 모델”입니다. PTQ보다 손이 더 가지만, 정확도 손실을 줄이고 싶을 때 강력합니다.

Preserving QAT: 조합 경량화

대응 노트북:

  • EX_09_Cluster_Preserving_QAT.ipynb
  • EX_10_Sparsity_Preserving_QAT.ipynb
  • EX_11_Sparsity_and_Cluster_Preserving%20QAT.ipynb

핵심 개념

Pruning, Clustering, Quantization은 따로 쓸 수도 있지만 함께 쓸 수도 있습니다. 문제는 단순히 이어 붙이면 앞 단계에서 만든 특성이 다음 단계에서 깨질 수 있다는 점입니다.

아주 쉽게 말하면 Preserving QAT는 “앞에서 열심히 줄여 놓은 특징을 QAT 중에도 망가뜨리지 않게 지키는 방법”입니다.

쉬운 비유

가방을 정리했다고 생각해 봅니다.

  • Pruning: 안 쓰는 물건을 뺐습니다.
  • Clustering: 비슷한 물건끼리 봉지에 묶었습니다.
  • QAT: 작은 가방에 다시 넣는 연습을 합니다.

그런데 다시 넣는 과정에서 물건이 흐트러지면 앞에서 정리한 의미가 줄어듭니다. Preserving QAT는 “정리한 상태를 유지하면서 다시 넣는 방법”입니다.

예를 들어:

  • Clustering으로 대표값 8개만 쓰게 만들었다.
  • 그런데 일반 QAT를 적용하면서 가중치가 다시 여러 값으로 퍼질 수 있다.

Preserving QAT는 이런 특성을 보존하면서 QAT를 적용하는 방식입니다.

EX_09: Cluster Preserving QAT

먼저 clustering을 적용합니다.

clustering_params = {
    "number_of_clusters": 8,
    "cluster_centroids_init": tfmot.clustering.keras.CentroidInitialization.KMEANS_PLUS_PLUS,
    "cluster_per_channel": True
}

clustered_model = tfmot.clustering.keras.cluster_weights(model, **clustering_params)

그 다음 일반 QAT와 cluster-preserving QAT를 비교합니다.

quant_aware_annotate_model = tfmot.quantization.keras.quantize_annotate_model(
    stripped_clustered_model
)

cqat_model = tfmot.quantization.keras.quantize_apply(
    quant_aware_annotate_model,
    tfmot.experimental.combine.Default8BitClusterPreserveQuantizeScheme()
)

핵심은 Default8BitClusterPreserveQuantizeScheme()입니다. QAT를 하면서 clustering 특성이 유지되도록 돕습니다.

EX_10: Sparsity Preserving QAT

Pruning으로 만든 sparse 특성을 QAT에서도 유지합니다.

pqat_model = tfmot.quantization.keras.quantize_apply(
    quant_aware_annotate_model,
    tfmot.experimental.combine.Default8BitPrunePreserveQuantizeScheme()
)

핵심 질문:

  • 일반 QAT 후에도 0 가중치가 많이 남아 있는가?
  • PQAT 후에는 sparsity가 더 잘 유지되는가?
  • gzip 또는 zip 압축 크기가 더 줄어드는가?

EX_11: Sparsity and Cluster Preserving QAT

이 예제는 pruning, clustering, QAT를 모두 엮습니다.

흐름:

baseline model
-> pruning
-> strip_pruning
-> clustering
-> strip_clustering
-> preserving QAT
-> TFLite 변환

초보자에게는 가장 복잡한 예제입니다. 처음 볼 때는 코드 전체를 외우려 하지 말고 “앞 단계의 특성을 다음 단계에서도 유지하려고 별도 scheme을 쓴다”는 점만 잡으면 됩니다.

이 노트를 처음 읽을 때 목표

이 노트는 한 번에 완전히 이해하기 어렵습니다. 처음에는 아래 세 문장만 이해하면 충분합니다.

  1. Pruning은 0을 많이 만드는 기법입니다.
  2. Clustering은 대표값 개수를 줄이는 기법입니다.
  3. Preserving QAT는 QAT 후에도 0과 대표값 특성이 유지되게 돕습니다.

일반 QAT와 Preserving QAT 차이

구분 일반 QAT Preserving QAT
목적 양자화 오차 적응 양자화하면서 sparsity/cluster 유지
사용 API quantize_model quantize_annotate_model + quantize_apply
장점 코드가 단순 압축 특성 보존에 유리
단점 기존 경량화 특성이 깨질 수 있음 코드가 복잡

실습에서 확인할 것

  • QAT model과 CQAT/PQAT model의 cluster 수 또는 sparsity가 다른가?
  • 압축 크기 비교에서 preserving 방식이 유리한가?
  • accuracy 손실이 허용 가능한가?
  • 실전 배포에서는 복잡도 대비 이득이 충분한가?

초보자 포인트

Preserving QAT는 “경량화 특성을 지키면서 QAT를 하는 방법”입니다. 여러 기법을 조합할 때 필요한 고급 단계입니다.

real_value = scale * (quantized_value - zero_point)

온디바이스 AI 경량화 최신 트렌드

먼저 한 문장으로

최근 온디바이스 AI 경량화의 핵심은 “모델 성능을 더 높이는 것”보다 “성능을 크게 잃지 않으면서 작은 기기에서 실제로 돌아가게 만드는 것”입니다.

서버에서는 모델이 크면 GPU를 더 붙일 수 있습니다. 하지만 라즈베리파이, 스마트폰, 노트북, 엣지 카메라 같은 기기에서는 메모리, 전력, 발열, 응답 시간이 모두 제한됩니다.

온디바이스 AI의 현실 제약
= 작은 메모리
+ 낮은 연산 성능
+ 배터리와 발열 제한
+ 네트워크 없이 동작해야 하는 상황
+ 실시간 응답 요구

그래서 경량화의 성공 기준은 다음에 가깝습니다.

정확도 유지
+ 모델 크기 감소
+ 메모리 사용량 감소
+ 추론 지연시간 감소
+ 전력/발열 감소
+ 로컬 실행 가능

이 강의 실습과 최신 흐름

이 강의의 MNIST/RPS 실습은 CNN 기반 경량화의 기본기를 다룹니다.

이 강의 실습 배우는 핵심
Pruning: 가지치기 중요하지 않은 연결을 0으로 만들기
Clustering: 가중치 군집화 비슷한 가중치를 대표값으로 묶기
PTQ: 학습 후 양자화 학습 후 숫자 표현을 작게 바꾸기
QAT: 양자화 인지 학습 양자화될 상황을 미리 알고 학습하기
RPS 실전 이미지 분류 QAT 실제 이미지 분류 모델을 TFLite로 배포하기

최근 Hugging Face와 arXiv의 이슈는 이 개념이 LLM/VLM으로 확장된 형태입니다.

CNN 시대:
Pruning, Clustering, INT8 TFLite, FLOAT16 TFLite

LLM/VLM 시대:
4-bit quantization, GGUF, QLoRA, KV cache compression, local runtime 최적화

트렌드 1. Weight quantization은 기본 기술이 됨

Hugging Face Transformers 문서는 quantization을 “모델 weight를 낮은 precision으로 저장해 메모리 요구량을 낮추면서 정확도를 최대한 보존하는 방법”으로 설명합니다.

초보자식으로 말하면 다음과 같습니다.

float32: 숫자 하나를 32비트로 저장
float16: 숫자 하나를 16비트로 저장
int8:    숫자 하나를 8비트로 저장
int4:    숫자 하나를 4비트로 저장

온디바이스에서는 가장 먼저 이 질문을 봅니다.

이 모델이 기기 메모리에 올라가는가?

정확도가 조금 좋아도 기기 메모리에 올라가지 않으면 사용할 수 없습니다. 그래서 최근에는 8-bit뿐 아니라 4-bit, 3-bit, 2-bit, 1.58-bit 같은 더 낮은 bit 표현도 활발히 연구됩니다.

관련 자료:

  • Hugging Face Transformers Quantization Overview: https://huggingface.co/docs/transformers/main/en/quantization/overview
  • Hugging Face bitsandbytes 문서: https://huggingface.co/docs/transformers/main/en/quantization/bitsandbytes

트렌드 2. QLoRA와 parameter-efficient fine-tuning

온디바이스 관점에서 fine-tuning은 어렵습니다. 작은 기기에서 직접 학습하기에는 메모리와 연산량이 부족하기 때문입니다.

==실전에서는== 보통 다음 흐름을 씁니다.

서버 또는 GPU 환경에서 fine-tuning
-> quantization 또는 경량화
-> 온디바이스용 포맷으로 변환
-> 기기에서는 추론만 수행

LLM에서는 QLoRA가 대표적인 실용 방식입니다. 큰 base model은 4-bit로 줄여 메모리를 아끼고, 작은 LoRA adapter만 학습합니다.

전체 모델을 다시 학습하지 않음
-> 작은 adapter만 학습
-> 학습 비용 감소
-> 배포 시 quantized model과 adapter 조합

이것은 이 강의의 QAT와 직접 같지는 않지만, “경량화된 상태에서도 성능 손실을 줄이기 위해 추가 학습을 설계한다”는 점에서 연결됩니다.

트렌드 3. GGUF와 llama.cpp 계열이 로컬 실행의 중심이 됨

CNN 실습에서는 TFLite가 온디바이스 배포 포맷입니다.

LLM 로컬 실행에서는 GGUF가 매우 중요합니다. GGUF는 llama.cpp 계열에서 널리 쓰이는 모델 파일 포맷이고, 다양한 quantized data type을 지원합니다.

Hugging Face 모델
-> GGUF quantized 모델
-> llama.cpp / local runtime
-> 노트북, 맥, 미니 PC, 라즈베리파이 같은 로컬 기기에서 실행

온디바이스 LLM 실습을 한다면 TFLite뿐 아니라 GGUF도 함께 알아두면 좋습니다.

관련 자료:

  • Hugging Face GGUF 문서: https://huggingface.co/docs/transformers/main/en/gguf

트렌드 4. KV cache가 새로운 메모리 병목이 됨

CNN은 이미지 한 장을 넣고 결과 하나를 받으면 끝입니다.

LLM은 다릅니다. 문장을 한 토큰씩 생성하면서 이전 문맥을 계속 기억해야 합니다. 이때 key-value cache, 줄여서 KV cache가 커집니다.

짧은 질문:
모델 weight 메모리가 중요

긴 대화:
모델 weight + KV cache 메모리가 중요

문맥 길이가 길어지고 대화가 길어질수록 KV cache가 큰 문제가 됩니다. 그래서 최근 arXiv에서는 weight quantization뿐 아니라 KV cache quantization, KV cache compression이 큰 주제입니다.

예를 들어 adaptive KV-cache quantization 연구는 token 중요도에 따라 2-bit, 4-bit, 8-bit, FP16을 동적으로 배정합니다.

쉬운 비유:

중요한 내용은 선명하게 저장
덜 중요한 내용은 흐리게 저장
전체 메모리는 줄이고 답변 품질은 유지

관련 논문:

  • Don’t Waste Bits! Adaptive KV-Cache Quantization for Lightweight On-Device LLMs: https://arxiv.org/abs/2604.04722
  • SVDq: 1.25-bit and 410x Key Cache Compression for LLM Attention: https://arxiv.org/abs/2502.15304
  • Unifying KV Cache Compression for Large Language Models with LeanKV: https://arxiv.org/abs/2412.03131

트렌드 5. 1.58-bit와 ternary LLM

최근 화제 중 하나는 1.58-bit LLM입니다. weight를 세 가지 값으로 표현합니다.

-1, 0, +1

세 가지 상태를 표현하는 데 필요한 정보량이 log2(3), 즉 약 1.58 bit라서 1.58-bit라고 부릅니다.

이 방식은 단순히 학습이 끝난 모델을 int8로 바꾸는 PTQ와 다릅니다. 모델 구조와 학습 방식 자체가 달라질 수 있습니다. Hugging Face 블로그도 BitNet 계열의 1.58-bit fine-tuning을 다루며, 극단적 양자화는 쉬운 변환 문제가 아니라 학습 설계 문제라는 점을 보여줍니다.

핵심 메시지:

PTQ:
다 만든 모델을 나중에 작게 바꿈

QAT:
작게 바뀔 것을 미리 알고 학습

1.58-bit LLM:
처음부터 아주 낮은 bit 표현을 전제로 구조와 학습을 설계

관련 자료:

  • Hugging Face, Fine-tuning LLMs to 1.58bit: https://huggingface.co/blog/1_58_llm_extreme_quantization

트렌드 6. 경량화와 runtime 최적화가 함께 감

모델 파일만 작다고 실제 속도가 빨라지는 것은 아닙니다.

이 강의에서 pruning을 배울 때도 같은 이야기를 했습니다.

0이 많아졌다고 무조건 빨라지지 않음
-> runtime이 sparse 연산을 잘 지원해야 함

LLM에서도 마찬가지입니다.

모델 quantization
+ KV cache 관리
+ batching
+ offloading
+ local runtime
+ 하드웨어 가속
= 실제 온디바이스 성능

즉, 최근 경량화는 알고리즘만의 문제가 아니라 시스템 문제입니다.

지연시간이 짧은가? 매우 중요
발열과 전력 소모가 괜찮은가? 매우 중요
네트워크 없이 동작하는가? 중요
개인정보가 기기 밖으로 나가지 않는가? 중요

그래서 온디바이스 AI에서 경량화는 성능을 올리는 기술이라기보다, 성능을 크게 잃지 않으면서 작은 기기에서 실행 가능하게 만드는 기술입니다.

경량화 성공 = 정확도 상승
이 아니라
경량화 성공 = 정확도 유지 + 모델 크기 감소 + 기기 실행 가능

그리고 최신 트렌드까지 연결하면 다음과 같습니다.

1단계: CNN 모델을 TFLite INT8/FLOAT16으로 줄여보기
2단계: 정확도와 파일 크기 비교하기
3단계: 라즈베리파이에서 실제 추론 시간 측정하기
4단계: LLM으로 확장할 때 GGUF와 KV cache 문제가 등장함을 이해하기

이 실습에서 얻어야 할 감각

실습에서 실제 성능 개선이 크게 보이지 않을 수도 있습니다. 또한 경량화를 해도 정확도 저하가 크지 않을 수 있습니다.

이 결과는 실패가 아닙니다. 오히려 중요한 메시지입니다.

경량화는 정확도를 올리는 마법이 아니다.
정확도를 가능한 유지하면서 배포 가능한 모델로 만드는 조정 과정이다.

Fine-tuning도 마찬가지입니다. 단순히 몇 epoch 더 학습한다고 항상 좋아지지 않습니다. learning rate, dataset, layer freezing, quantization 방식, target hardware가 모두 영향을 줍니다.

온디바이스 AI에서 fine-tuning은 어렵고, 그래서 최근 연구는 다음 방향으로 갑니다.

  • 쉬운 PTQ부터 먼저 시도
  • 정확도 손실이 크면 QAT 또는 LoRA/QLoRA 사용
  • weight뿐 아니라 KV cache까지 압축
  • 모델 포맷과 runtime까지 함께 최적화
  • 실제 기기에서 latency, memory, power를 측정

피지컬 AI 시대: On-device AI가 더 중요해지는 이유

최근 AI 트렌드는 텍스트를 잘 생성하는 LLM을 넘어, 현실 세계에서 보고 판단하고 움직이는 피지컬 AI로 확장되고 있습니다.

피지컬 AI는 다음을 동시에 처리해야 합니다.

Vision: 카메라와 센서로 현실을 본다
Language: 사람의 명령과 상황을 이해한다
Action: 로봇 팔, 바퀴, 손, 드론을 움직인다

그래서 최신 로봇 AI에서는 VLA, 즉 Vision-Language-Action 모델이 중요한 흐름입니다.

이미지/영상 입력
+ 자연어 명령
+ 로봇 행동 출력
= VLA 모델

대표적인 최신 흐름은 다음과 같습니다.

흐름 의미 온디바이스와의 관계
VLA 모델 시각, 언어, 행동을 하나의 모델로 연결 로봇이 현장에서 보고 명령을 이해하고 행동해야 함
로봇 foundation model 다양한 로봇 작업을 하나의 큰 모델로 일반화 큰 모델을 작게 줄여 로봇 안에서 실행해야 함
World model 현실 세계의 변화를 예측하는 모델 실제 실험 비용과 위험을 줄이고 시뮬레이션 학습에 활용
Sim-to-real 시뮬레이션에서 배운 것을 실제 로봇에 적용 실제 로봇 데이터 부족 문제를 완화
On-device robot AI 클라우드 없이 로봇 내부에서 추론 지연시간, 안전성, 네트워크 단절 문제 해결

이 흐름에서 On-device AI는 선택이 아니라 필수에 가깝습니다.

로봇은 현실에서 즉시 반응해야 한다.
네트워크가 끊겨도 동작해야 한다.
사람 옆에서 안전하게 움직여야 한다.
센서 데이터가 외부로 나가지 않아야 할 수 있다.

클라우드 AI는 강력하지만, 피지컬 AI에서는 다음 문제가 생깁니다.

네트워크 지연
연결 불안정
개인정보와 현장 데이터 보안
서버 비용
긴급 상황에서 늦은 반응

따라서 피지컬 AI의 실전 구조는 보통 다음 방향으로 갑니다.

큰 서버/GPU에서 학습
-> 시뮬레이션과 실제 데이터로 검증
-> 작은 on-device 모델로 distillation / quantization
-> 로봇 내부에서 실시간 추론

피지컬 AI 최신 트렌드와 경량화의 연결

피지컬 AI에서는 모델만 똑똑하면 부족합니다. 현실에서 움직이기 위해서는 전체 시스템이 가벼워야 합니다.

센서 입력
-> 인식 모델
-> 상황 이해
-> 행동 계획
-> 제어 출력

이 모든 단계가 짧은 시간 안에 끝나야 합니다.

단계 경량화 이슈
센서 입력 영상/이벤트 데이터량을 줄여야 함
인식 모델 CNN, ViT, VLA 모델을 작게 만들어야 함
문맥 이해 LLM/VLM의 weight와 KV cache를 줄여야 함
행동 계획 지연시간이 짧아야 함
제어 출력 실시간성과 안전성이 중요

즉, 피지컬 AI의 경량화는 단순히 float32 -> int8만이 아닙니다.

모델 경량화
+ 센서 데이터 경량화
+ runtime 최적화
+ 하드웨어 가속
+ 안전한 실시간 제어
= 피지컬 AI용 On-device AI

김임환 교수의 연구분야 “DVS” : 센서 단계에서 시작하는 경량화

DVS, Dynamic Vision Sensor는 피지컬 AI와 온디바이스 AI에서 매우 중요한 센서 기술입니다.

일반 카메라는 매 순간 전체 프레임을 찍습니다.

일반 카메라:
전체 화면을 30fps 또는 60fps로 계속 촬영

DVS는 다릅니다. 밝기 변화가 생긴 픽셀만 이벤트로 출력합니다.

DVS:
변화가 생긴 픽셀만 비동기 이벤트로 출력

이 차이는 온디바이스 피지컬 AI에서 매우 큽니다.

일반 카메라 DVS

쉽게 말하면 DVS는 입력 단계에서 이미 경량화를 하는 센서입니다.

왜 DVS가 피지컬 AI에 잘 맞는가?

피지컬 AI는 현실 세계에서 빠르게 움직이는 대상을 봐야 합니다.

예를 들어:

  • 드론이 빠르게 이동할 때
  • 로봇 팔이 물체를 집을 때
  • 자율주행 시스템이 갑자기 튀어나오는 물체를 볼 때
  • 산업 설비에서 작은 이상 움직임을 감지할 때
  • 웨어러블/AR 기기가 낮은 전력으로 주변 변화를 감지할 때

이런 상황에서는 전체 프레임을 느리게 처리하는 것보다, 변화가 생긴 부분을 즉시 감지하는 것이 중요합니다.

분야 DVS가 유리한 이유
로봇 손/팔 제어 빠른 접촉과 움직임 감지
드론 고속 이동, 저전력, motion blur 감소
자율주행 빠른 물체 움직임과 넓은 dynamic range
산업 검사 미세 변화와 빠른 이상 탐지
웨어러블/AR 저전력 시각 인식
뉴로모픽 컴퓨팅 이벤트 기반 sparse 연산과 구조적으로 잘 맞음

취리히 드론 프로젝트 사례: DVS가 왜 피지컬 AI에 중요한가?

취리히권 연구에서 DVS와 드론을 결합한 사례는 매우 좋은 강의 사례입니다. 흔히 “취리히공대 드론 연구”로 묶어 말하지만, 정확히는 University of Zurich의 Robotics and Perception Group, ETH Zurich 출신 연구자, Davide Scaramuzza 연구 그룹을 포함한 취리히 로보틱스 생태계의 성과로 보는 것이 좋습니다.

핵심 메시지는 다음과 같습니다.

빠르게 움직이는 드론은 늦게 보면 이미 늦다.
그래서 프레임 카메라보다 지연시간이 낮은 DVS가 유리하다.

대표 사례 1. Event camera로 드론 동적 장애물 회피

EVDodgeNet은 단일 event camera와 온보드 계산으로 쿼드로터가 움직이는 장애물을 피하는 연구입니다. 논문은 동적 장애물 회피에는 낮은 지연시간이 필요하고, event camera가 이런 상황에 적합하다고 설명합니다. 또한 얕은 신경망들을 사용해 자기 움직임과 독립적으로 움직이는 물체를 추정하고, 시뮬레이션에서 학습한 뒤 실제 환경으로 전이하는 방식을 보입니다.

관련 링크:

  • EVDodgeNet: Deep Dynamic Obstacle Dodging with Event Cameras: https://arxiv.org/abs/1906.02919
  • Dynamic obstacle avoidance for quadrotors with event cameras, Science Robotics 2020 관련 정보: https://www.science.org/doi/10.1126/scirobotics.aaz9712
  • IEEE Spectrum 관련 기사/영상 검색: https://www.youtube.com/results?search_query=Event+Camera+Helps+Drone+Dodge+Thrown+Objects

강의에서 강조할 점:

일반 카메라:
프레임을 기다림
빠른 공이나 장애물은 motion blur 발생 가능

DVS:
변화 이벤트가 즉시 발생
빠른 물체의 움직임을 낮은 지연시간으로 감지

대표 사례 2. Monocular event camera 기반 쿼드로터 장애물 회피

2024년 연구 Monocular Event-Based Vision for Obstacle Avoidance with a Quadrotor는 단일 온보드 event camera만으로 쿼드로터 정적 장애물 회피를 시도한 사례입니다. 흥미로운 점은 전통적인 카메라 기반 회피와 달리, 빠른 움직임에서 event-based depth estimation이 더 잘 동작할 수 있다는 관찰입니다.

관련 링크:

  • Monocular Event-Based Vision for Obstacle Avoidance with a Quadrotor: https://arxiv.org/abs/2411.03303

강의에서 강조할 점:

DVS는 가만히 있는 장면보다 움직임이 있는 장면에서 정보가 많이 나온다.
그래서 빠르게 움직이는 드론, 로봇, 차량에서 강점이 커진다.

이 드론 프로젝트들은 피지컬 AI에서 DVS가 왜 중요한지 잘 보여줍니다.

피지컬 AI 요구 DVS 드론 사례에서의 의미
초저지연 빠르게 날아오는 물체를 즉시 감지
저전력 전체 프레임을 계속 처리하지 않음
온보드 계산 클라우드가 아니라 드론 내부에서 판단
빠른 움직임 대응 motion blur에 강함
sim-to-real 시뮬레이션 학습 후 실제 드론 적용 가능

이것은 단순히 센서 성능이 좋은 사례가 아닙니다. 온디바이스 AI 전체 구조를 보여주는 사례입니다.

DVS 센서
-> sparse event stream
-> 작은/빠른 인식 모델
-> 온보드 추론
-> 실시간 드론 제어

따라서 DVS 드론 연구는 피지컬 AI 시대의 핵심 질문에 직접 연결됩니다.

현실 세계에서 빠르게 변하는 정보를
어떻게 작은 기기 안에서
낮은 전력과 낮은 지연시간으로 처리할 것인가?

연구 포지셔닝

Physical AI를 위한 event-based on-device perception
DVS 기반 초저지연, 저전력 온디바이스 시각 지능
피지컬 AI 시대에는 모델만 경량화해서는 부족하다.
센서 입력부터 sparse하고 event-driven하게 만들어야 한다.
DVS는 이 방향의 핵심 센서 기술이다.

DVS경량화는 주로 모델 내부를 줄이는 방법입니다.

Pruning: 연결을 줄임
Clustering: 값의 종류를 줄임
PTQ: 숫자 저장 크기를 줄임
QAT: 양자화 후 정확도 손실을 줄임

DVS는 이보다 앞단에서 시작합니다.

DVS:
처리할 입력 자체를 줄임

그래서 DVS는 온디바이스 AI 경량화의 확장판으로 볼 수 있습니다.

센서 경량화
+ 이벤트 기반 인식
+ 온디바이스 추론
+ 피지컬 AI 실시간성
+ 저전력 엣지 컴퓨팅

핵심

DVS는 피지컬 AI에서 온디바이스 인식을 가능하게 하는 핵심 센서이다.
일반 카메라가 모든 프레임을 처리하는 반면,
DVS는 변화가 발생한 이벤트만 출력하므로 데이터량, 지연시간, 전력 소모를 줄일 수 있다.
따라서 DVS는 경량화가 모델 내부에서만 이루어지는 것이 아니라
센서 입력 단계에서 시작될 수 있음을 보여준다.
